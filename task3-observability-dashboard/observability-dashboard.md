## Overview

I used the existing Grafana dashboard from the Docker Compose environment and extended it with additional monitoring panels.

The dashboard helps monitor:
- application health
- API performance
- database activity
- Kafka processing
- infrastructure usage

The dashboard was tested locally using real API requests and Kafka events.

---

# Panels Included

## JVM Heap Memory Used
Shows Java heap memory usage.

Useful for:
- detecting memory leaks
- monitoring JVM health

---

## HTTP Request Rate
Shows incoming requests per second.

Useful for:
- traffic monitoring
- detecting spikes

---

## HTTP 5xx Rate by Path
Shows server-side errors grouped by endpoint.

Useful for:
- identifying failing APIs
- troubleshooting production issues

This panel detected the `/quote/segment` 500 error during testing.

---

## Application Logs
Shows logs collected by Loki.

Useful for:
- debugging
- root cause analysis

This panel helped identify the Kafka deserialization issue.

---

## Pricing DB Reads / sec
Shows database reads generated by pricing operations.

Useful for:
- identifying expensive requests
- detecting database-heavy processing

---

## DB Read Duration p95
Shows the 95th percentile database latency.

Useful for:
- detecting slow database operations
- identifying bottlenecks

---

## Controller Request p95
Shows API response latency.

Useful for:
- identifying slow endpoints
- monitoring user experience

This panel showed high latency during the heavy quote request test.

---

## Process CPU Usage
Shows CPU usage of the Spring Boot process.

Useful for:
- autoscaling decisions
- detecting CPU-intensive operations

Thresholds:
- Warning: 70%
- Critical: 90%

---

## Kafka Listener Messages / sec
Shows Kafka consumer activity.

Useful for:
- monitoring Kafka processing
- detecting consumer issues

This panel was tested by publishing pricing events.

---

## Active DB Connections
Shows active PostgreSQL connections.

Useful for:
- detecting connection pool exhaustion
- monitoring database pressure

---

# Most Important Health Signals

The most important signals for this service are:
- HTTP 5xx errors
- API latency
- database latency
- Kafka consumer activity
- database connections
- application logs

---

# Dashboard Validation

The dashboard was tested using:
- slow quote requests
- Kafka price events
- failing endpoints

Observed results:
- database reads increased during heavy requests
- API latency increased significantly
- Kafka activity appeared correctly
- logs captured processing failures
